# Custom Pass Shaders (Render Graph)

This guide covers custom pass shaders inside render graph pass definitions.

## 1. Where Pass Shaders Live

Custom pass shaders are declared in render graph resources (`cmd-render-graph-upsert`) as part of pass spec.

The runtime compiles pass logical shader source into physical WGSL.

## 2. Logical Shader Shape

The generated wrapper expects your source to define:

- `fn fragment(input: FragmentInput) -> FragmentOutput`

For screen passes, vertex/fullscreen triangle is generated by the core.

## 3. Built-in Frame Semantics in Pass DSL

Physical WGSL generation injects:

- `sample_scene_color(uv)`
- `sample_history0(uv)`
- `sample_history1(uv)`
- `load_scene_depth(pixel)`
- `scene_resolution()`
- `scene_inv_resolution()`
- `current_frame_index()`

These are backed by runtime resources bound by the engine.

## 4. Params Uniform

If pass shader `params` are declared in graph spec, a `params` uniform struct is auto-generated in group 0 binding 0.

Use parameter names exactly as declared in pass spec.

## 5. Inputs/Outputs Contract

- Output fields in `FragmentOutput` must match declared pass outputs.
- For declared inputs, helper function names are generated:
  - `sample_<input_name>(uv)`
  - `load_<input_name>(pixel)`

Current runtime behavior for generic logical `sample_/load_` inputs is placeholder-based unless explicitly wired by pass implementation.

## 6. Example: Temporal Blend Pass

```wgsl
fn fragment(input: FragmentInput) -> FragmentOutput {
  var out: FragmentOutput;
  let uv = input.uv;
  let curr = sample_scene_color(uv);
  let hist = sample_history0(uv);
  let blended = mix(curr.rgb, hist.rgb, 0.35);
  out.color = vec4<f32>(blended, 1.0);
  return out;
}
```

## 7. Validation Tips

- Keep pass output names aligned with graph resource IDs.
- Prefer deterministic math and explicit clamping.
- If pass uses history, also ensure material/pass capabilities and runtime path keep history active.
